# 键道项目经验记录

记录日期：2026-07-04

## 本次完成的内容

- 完成打字练习主界面优化：拼音和汉字排版更整齐，练习内容支持向下滚动，删除了底部多余线条，键盘区域下移，整体界面更干净。
- 去掉普通中文练习中“必须按空格”的要求。完整一句话里没有实际空格时，不需要按空格；只有文本本身真的有空格时才需要输入空格。
- 添加唐诗三百首栏目，进入练习后按空格可切换下一首。每首诗显示标题和作者，但标题、作者不参与练习。
- 添加菜根谭全集栏目，逻辑和唐诗一样：分篇练习，标题/作者信息不进入练习范围，空格切换下一篇。
- 支持导入 txt/text 文件。导入文章后自动配置拼音。
- 针对新概念英语第一册 OCR 文本做了兼容：能从较大的 text/txt 文件中提取英文正文并切分为练习内容。
- 增强删除按钮：
  - 基础练习、英语新概念等内置栏目可以隐藏/删除栏目；
  - 我的文章可以删除当前自定义文章；
  - 支持恢复被隐藏的默认栏目。

## 成功经验

- 先把界面问题和数据问题分开处理：界面负责显示、滚动、键盘布局；数据负责栏目、文章、标题作者、拼音生成。
- 唐诗三百首和菜根谭这种全集内容，不要把所有正文一次铺满界面，应该做成“栏目 + 子文章”的结构。
- 标题和作者应该作为元信息显示，不放进练习正文。这样练习更纯净，也避免输入时被标题干扰。
- 拼音和汉字排版要按字或词的单元对齐，不能让拼音连续挤在一起。
- 大体量文章导入时，不能简单整文件塞进一个练习项，要清洗并分段/分篇。
- 改完后要做两类验证：
  - 代码语法检查；
  - 浏览器里实际看布局、滚动、栏目切换、删除、导入。

## 猜到并验证过的坑

- “拼音没有间隔”不是单纯字体大小问题，主要是每个字/拼音单元的布局宽度、间距和换行方式不合适。
- “汉字排列不整齐”通常是拼音行和汉字行没有被绑定成同一个显示单元，应该让一个汉字和它的拼音一起排版。
- “练习内容显示不全”不应该靠压缩字体解决，应该让练习内容区域可以滚动。
- “按空格”不能一刀切。中文整句练习里空格不是必需输入；但英文或文本真实含空格时，空格仍然是练习内容的一部分。
- PDF 转出来的新概念文本可能带页眉、页脚、乱码、重复内容、OCR 残留，导入时要容错。
- 删除按钮不能只删除“我的文章”，也要能处理内置栏目，否则用户会觉得删不掉。

## Git 推送特别记录

这个项目每次访问 Git/GitHub 都容易出连接或认证问题，后面一定先按下面顺序排查。

### 网络环境

- 用户使用 v2ray，但不是全局代理。
- 本机代理地址这次可用的是：
  - `http://127.0.0.1:10808`
- Git/GitHub 命令需要显式走代理，否则很容易连不上或超时。

### 本次遇到的问题

- `git push origin main` 触发 Windows 弹窗：
  - `git-remote-https.exe - Application Error`
  - `The memory could not be read`
- 这不是项目代码的问题，是 Git 的 HTTPS 远程传输/认证组件崩溃。
- GitHub CLI 也显示过令牌异常：
  - `The token in keyring is invalid`
- `gh auth refresh`、`gh auth login`、Git Credential Manager 设备码登录都出现过卡住/超时。
- 但通过代理访问 GitHub 本身是通的，`curl -I https://github.com` 返回成功。

### 本次成功推送的方法

本地已经提交后，先确认：

```powershell
git status --short --branch
git log -1 --oneline
```

如果 `gh api repos/jfcws1/- --jq .full_name` 能成功，说明 GitHub CLI 至少有可用的 API 令牌。

然后使用 GitHub CLI 的 token 临时给 git push 授权，绕过崩溃的交互登录：

```powershell
$env:HTTPS_PROXY='http://127.0.0.1:10808'
$env:HTTP_PROXY='http://127.0.0.1:10808'
$env:ALL_PROXY='http://127.0.0.1:10808'
$env:GIT_TERMINAL_PROMPT='0'
$env:GIT_ASKPASS=''
$token = gh auth token
$pair = "x-access-token:$token"
$basic = [Convert]::ToBase64String([Text.Encoding]::ASCII.GetBytes($pair))
git -c credential.helper= -c core.askPass= -c http.proxy=http://127.0.0.1:10808 -c https.proxy=http://127.0.0.1:10808 -c http.https://github.com/.extraheader="AUTHORIZATION: basic $basic" push origin main
```

本次最终成功推送：

```text
5d2c202 Build typing practice content library
```

### 下次优先做

1. 先检查代理是否通：

```powershell
$env:HTTPS_PROXY='http://127.0.0.1:10808'
$env:HTTP_PROXY='http://127.0.0.1:10808'
curl.exe -I https://github.com --max-time 20
```

2. 再检查仓库状态：

```powershell
git status --short --branch
git remote -v
```

3. 如果普通 `git push` 又弹 `git-remote-https.exe` 崩溃，不要反复点重试，直接用“临时 Basic Token 头”的成功方法推送。

4. 推送成功后确认：

```powershell
git status --short --branch
git log -1 --oneline
```

看到 `## main...origin/main` 且没有 `[ahead 1]`，说明已经同步到 GitHub。

## 本轮指法练习与成品体验复盘

### 本轮成功完成的内容

- 将项目名称和产品方向稳定为“键道”，避免继续沿用别人的“打字侠”名称。
- 为指法练习增加固定栏目，且不可删除，作为软件核心训练入口。
- 指法练习包含字母、数字、符号和进阶组合练习，形成从基础到进阶的训练路径。
- 增加键盘手指虚影：先做指尖提示，再加入整手模式。
- 将正确起手姿势明确到键盘上：
  - 左手小指、无名指、中指、食指分别放在 `A S D F`；
  - 右手食指、中指、无名指、小指分别放在 `J K L ;`；
  - 两个拇指分别轻放在空格键左右位置。
- 修复指尖和整手虚影位置漂移问题，让手指位置根据真实按键中心计算，而不是靠固定像素猜。
- 增强整手模式动作：上排、下排、符号键输入时，对应手指会移动到目标键，不再只停在中排。
- 增强按键反馈：用户实际敲下的键有更明显的短暂绿色反馈，避免被“下一键”蓝色提示干扰。
- 完成时冻结成绩：练习完成瞬间记录完成时间，用时、速度、准确率不再继续增长。
- 增加顶部练习控制：暂停、重来、音效开关、指尖/整手、全屏。
- 优化手指虚影可见度：基准手位不再过淡，整手轮廓更容易看清。

### 成功经验

- 视觉问题不能只靠“感觉调像素”。凡是和键盘位置绑定的东西，都应该读取真实按键的 `getBoundingClientRect()`，用真实中心点计算。
- 指法训练不能只显示“下一个键”，还必须显示“手应该放在哪里”。基础姿势是教学的一部分，不是装饰。
- 手指提示要分两层：
  - 常驻基准手位，告诉用户手应该放在哪里；
  - 当前动作提示，告诉用户哪个手指去按哪个键。
- “下一键提示”和“用户已按键反馈”要有明显视觉层级。下一键可以蓝色常驻，用户刚敲的键应该短暂强反馈。
- 计时类功能必须有清晰状态机：未开始、进行中、暂停、已完成。完成后必须封存成绩，不能继续读取当前时间。
- 用户截图非常关键。很多偏移问题只看代码不明显，一张截图能直接暴露实际视觉问题。
- 每次小修都及时跑：
  - `new Function(script)` 脚本语法检查；
  - `git diff --check` 空白和基本 diff 检查；
  - `git status --short` 确认工作区状态。
- 每次稳定后及时提交并推送，避免多轮 UI 调整混在一个不可回溯的大改里。

### 踩过的坑

- 一开始把手指虚影当成“画一个手”的问题，实际它是“教学姿势映射”问题。没有先定义 `A S D F / J K L ; / Space`，就容易越画越偏。
- 固定像素坐标很危险。顶部统计栏、窗口尺寸、键盘宽度变化后，手指层会和键盘错位。
- 只让指尖高亮还不够。整手模式如果整根手指不动，用户会觉得呆板，也不符合“伸指按键”的真实动作。
- 空格不能只提示一个拇指。正确姿势是双拇指都轻放在空格上，视觉上也应该表达出来。
- “完成成绩”如果没有 `finishTime`，计时器会继续增长。这个 bug 表面上是 UI 文案问题，本质是状态没有封存。
- 按键反馈太弱时，用户会误以为“敲的键没亮”。尤其下一键高亮和基准键虚影同时存在时，真实输入反馈必须更强。
- 本地文件在 Codex 的浏览器工具里不一定能直接访问，`file://` 和 `127.0.0.1` 有时会被环境限制。不能把“工具无法预览”误判为“代码无法运行”。
- PowerShell 默认输出可能显示中文乱码。读写中文文档时优先用 UTF-8 工具或 Node 读取，避免误以为文件坏了。

## 用户沟通复盘：需要继续保持的优点

- 你愿意连续测试、截图、指出具体问题，这非常重要。软件体验就是靠这种一轮一轮磨出来的。
- 你对“正确指法姿势”的坚持是对的。这里不是审美问题，而是产品教学是否专业的问题。
- 你能及时说“这个好了，不用改了”，这能让我收住，不把已经稳定的地方继续折腾坏。
- 你会提出真实使用感受，比如“看着很呆板”“太淡，看着费力”。这种感受类反馈对 UI 很有价值。

## 用户沟通中的不足和改进建议

- 有些问题最开始描述偏结果，没有描述“期望标准”。例如“指尖不在正确的位置上”，如果一开始补一句“正确标准是小指落在 A、无名指落在 S”，我会更快定位到基准手位，而不是先修活动指尖。
- 截图很有用，但最好同时说明：
  - 当前处于哪个模式：指尖还是整手；
  - 当前下一个键是什么；
  - 你期望哪个手指应该在哪里。
- 对交互问题，建议用“我做了什么、看到了什么、我期望什么”三段式表达。例如：
  - 我按了 `q`；
  - 现在只有中排 `a` 附近亮；
  - 我期望左小指向上移动到 `q`，并且 `q` 键短暂亮起。
- 如果一个问题包含多个现象，尽量拆开编号。比如“手指没动”和“敲的键不亮”是两个问题，虽然相关，但实现点不同。
- 对“像某个网站”这类参考，最好明确要借鉴哪一部分，不要只给整体链接。例如：只借鉴打字区下划线、右侧统计、顶部按钮、手指动画，其他不要照搬。
- 你有时会在问题已经很接近本质时才补充标准。比如正确指法姿势这次，如果一开始就把 `F/J 凸起横杠、A/S/D/F、J/K/L/;、双拇指空格` 作为验收标准，我们能少走两轮。
- 测试时如果能按“先基础、再边界”的顺序，会更快：
  1. 中排基准键；
  2. 上排字母；
  3. 下排字母；
  4. 数字；
  5. 符号；
  6. 空格；
  7. 错误输入；
  8. 完成成绩。
