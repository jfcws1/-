# 键道项目经验记录

记录日期：2026-07-04

## 本次完成的内容

- 完成打字练习主界面优化：拼音和汉字排版更整齐，练习内容支持向下滚动，删除了底部多余线条，键盘区域下移，整体界面更干净。
- 去掉普通中文练习中“必须按空格”的要求。完整一句话里没有实际空格时，不需要按空格；只有文本本身真的有空格时才需要输入空格。
- 添加唐诗三百首栏目，进入练习后按空格可切换下一首。每首诗显示标题和作者，但标题、作者不参与练习。
- 添加菜根谭全集栏目，逻辑和唐诗一样：分篇练习，标题/作者信息不进入练习范围，空格切换下一篇。
- 支持导入 txt/text 文件。导入文章后自动配置拼音。
- 针对新概念英语第一册 OCR 文本做了兼容：能从较大的 text/txt 文件中提取英文正文并切分为练习内容。
- 增强删除按钮：
  - 基础练习、英语新概念等内置栏目可以隐藏/删除栏目；
  - 我的文章可以删除当前自定义文章；
  - 支持恢复被隐藏的默认栏目。

## 成功经验

- 先把界面问题和数据问题分开处理：界面负责显示、滚动、键盘布局；数据负责栏目、文章、标题作者、拼音生成。
- 唐诗三百首和菜根谭这种全集内容，不要把所有正文一次铺满界面，应该做成“栏目 + 子文章”的结构。
- 标题和作者应该作为元信息显示，不放进练习正文。这样练习更纯净，也避免输入时被标题干扰。
- 拼音和汉字排版要按字或词的单元对齐，不能让拼音连续挤在一起。
- 大体量文章导入时，不能简单整文件塞进一个练习项，要清洗并分段/分篇。
- 改完后要做两类验证：
  - 代码语法检查；
  - 浏览器里实际看布局、滚动、栏目切换、删除、导入。

## 猜到并验证过的坑

- “拼音没有间隔”不是单纯字体大小问题，主要是每个字/拼音单元的布局宽度、间距和换行方式不合适。
- “汉字排列不整齐”通常是拼音行和汉字行没有被绑定成同一个显示单元，应该让一个汉字和它的拼音一起排版。
- “练习内容显示不全”不应该靠压缩字体解决，应该让练习内容区域可以滚动。
- “按空格”不能一刀切。中文整句练习里空格不是必需输入；但英文或文本真实含空格时，空格仍然是练习内容的一部分。
- PDF 转出来的新概念文本可能带页眉、页脚、乱码、重复内容、OCR 残留，导入时要容错。
- 删除按钮不能只删除“我的文章”，也要能处理内置栏目，否则用户会觉得删不掉。

## Git 推送特别记录

这个项目每次访问 Git/GitHub 都容易出连接或认证问题，后面一定先按下面顺序排查。

### 网络环境

- 用户使用 v2ray，但不是全局代理。
- 本机代理地址这次可用的是：
  - `http://127.0.0.1:10808`
- Git/GitHub 命令需要显式走代理，否则很容易连不上或超时。

### 本次遇到的问题

- `git push origin main` 触发 Windows 弹窗：
  - `git-remote-https.exe - Application Error`
  - `The memory could not be read`
- 这不是项目代码的问题，是 Git 的 HTTPS 远程传输/认证组件崩溃。
- GitHub CLI 也显示过令牌异常：
  - `The token in keyring is invalid`
- `gh auth refresh`、`gh auth login`、Git Credential Manager 设备码登录都出现过卡住/超时。
- 但通过代理访问 GitHub 本身是通的，`curl -I https://github.com` 返回成功。

### 本次成功推送的方法

本地已经提交后，先确认：

```powershell
git status --short --branch
git log -1 --oneline
```

如果 `gh api repos/jfcws1/- --jq .full_name` 能成功，说明 GitHub CLI 至少有可用的 API 令牌。

然后使用 GitHub CLI 的 token 临时给 git push 授权，绕过崩溃的交互登录：

```powershell
$env:HTTPS_PROXY='http://127.0.0.1:10808'
$env:HTTP_PROXY='http://127.0.0.1:10808'
$env:ALL_PROXY='http://127.0.0.1:10808'
$env:GIT_TERMINAL_PROMPT='0'
$env:GIT_ASKPASS=''
$token = gh auth token
$pair = "x-access-token:$token"
$basic = [Convert]::ToBase64String([Text.Encoding]::ASCII.GetBytes($pair))
git -c credential.helper= -c core.askPass= -c http.proxy=http://127.0.0.1:10808 -c https.proxy=http://127.0.0.1:10808 -c http.https://github.com/.extraheader="AUTHORIZATION: basic $basic" push origin main
```

本次最终成功推送：

```text
5d2c202 Build typing practice content library
```

### 下次优先做

1. 先检查代理是否通：

```powershell
$env:HTTPS_PROXY='http://127.0.0.1:10808'
$env:HTTP_PROXY='http://127.0.0.1:10808'
curl.exe -I https://github.com --max-time 20
```

2. 再检查仓库状态：

```powershell
git status --short --branch
git remote -v
```

3. 如果普通 `git push` 又弹 `git-remote-https.exe` 崩溃，不要反复点重试，直接用“临时 Basic Token 头”的成功方法推送。

4. 推送成功后确认：

```powershell
git status --short --branch
git log -1 --oneline
```

看到 `## main...origin/main` 且没有 `[ahead 1]`，说明已经同步到 GitHub。
